# [Org_Code] 项目内文档语义分层标准 v1

> **协议编号**: 13-PROJECT-DOCUMENT-SEMANTICS  
> **版本**: v8.2 (Config-driven & Rubric-gated)  
> **状态**: ✅ 已生效  
> **生效日期**: 2026-03-23  
> **目标**: 在 00-04 顶层工作区结构之下，进一步定义项目目录内部的标准子目录语义，确保跨节点协作时能清楚区分过程文件、待审稿、定稿与归档。

---

## 一、背景

当前 [Org_Code] 已有顶层工作区分层：
- `00-Pending`
- `01-Projects`
- `02-Areas`
- `03-Resources`
- `04-Archive`

但项目目录内部的语义尚不统一，容易出现：
- 过程稿与定稿混放
- 审阅稿与最终交付物不易区分
- 跨节点引用时不清楚"哪个能信、哪个只是草稿"

因此，需要建立项目内部的标准目录语义。

---

## 二、标准子目录结构

```text
<project>/
├── README.md
├── INDEX.md
├── process_drafts/      # 过程推演 / 草稿 / 中间稿
├── working_notes/       # 过程笔记 / 讨论摘录 / 临时判断
├── review/              # 待审稿 / 审阅意见 / 对齐稿
├── final/               # 已通过 / 已定稿 / 高信任引用源
├── deliverables/        # 对外或对上交付物
└── archive/             # 项目内部历史归档
```

---

## 三、目录语义定义

### 1. `process_drafts/`
- 推演稿
- 草稿
- 中间稿
- **不得直接视为最终定案**

### 2. `working_notes/`
- 思路草记
- 讨论摘录
- 会议笔记
- 过程判断

### 3. `review/`
- 待审稿
- 有审阅意见但未最终定稿
- 临近成稿但尚未正式通过

### 4. `final/`
- 正式通过
- 当前项目的高信任文件
- 可作为跨节点正式引用依据

### 5. `deliverables/`
- 对外 / 对上交付物
- 若与 `final/` 有差异，需说明原因

### 6. `archive/`
- 历史版本
- 退役内容
- 保留但不作为当前真源

---

## 四、协作使用纪律

1. **跨节点引用时**，优先查看其所在子目录语义
2. **`process_drafts/` 默认按草稿处理**，不得直接当定案引用
3. **`final/` 才能默认作为高信任项目内真源**
4. **`deliverables/` 是交付视角**，不等于内部最终真源
5. **若目录语义与 frontmatter 状态冲突**，以更保守的判定为准，并要求复核

---

## 五、与元数据机制的关系

目录语义是**物理隔离层**，frontmatter 是**语义声明层**。

两者应共同发挥作用：
- 目录帮助人和 Agent 快速判断文件大类
- frontmatter 帮助系统进一步做机器可读判断

不应二选一。
