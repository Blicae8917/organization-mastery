## 一、核心理念

### 1.1 什么是 DCE Protocol

**Discuss → Confirm → Execute**

将 AI Agent 从"被动执行者"转变为"主动协作者"的安全协议。

### 1.2 解决的问题

- ❌ Agent 盲目执行，结果不符合预期
- ❌ 信息不足就开始行动
- ❌ 经验无法沉淀，每次从零开始
- ❌ 用户和 Agent 磨合效率低

### 1.3 核心创新

| 传统 DCE | DCE 2.0 |
|---------|---------|
| 一次性协议 | 持续学习系统 |
| 被动响应 | 主动信息挖掘 |
| 经验不沉淀 | 自动分层存储 |
| 固定流程 | 自适应优化 |

## 二、三层架构

### 2.1 协议层 (Layer 1) - DCE Core

#### D - Discuss（讨论）

**标准流程**:
1. 给出 2-3 个方案
2. 每个方案包含优劣分析
3. 主动挖掘信息（见第 3 章）

**触发信息挖掘的信号**:
- "我觉得..."
- "但是..."
- "能不能..."
- "如果..."
- 方案大幅调整

#### C - Confirm（确认）

**有效确认词**:
- "执行"
- "确认"
- "批准"
- "Proceed"

**无效确认词**（只是回应，不是批准）:
- "好的"
- "OK"
- "可以"

#### E - Execute（执行）

1. 再次确认执行范围
2. 执行并记录过程
3. 触发复盘机制

---

### 2.2 记忆层 (Layer 2) - Memory System

#### 分层存储架构

```
┌─────────────────────────────────────────┐
│ L0: 活跃经验 (SESSION-STATE.md)         │
│ 当前任务的临时调整，会话结束即丢弃         │
├─────────────────────────────────────────┤
│ L1: 短期经验 (memory/lessons/)          │
│ 最近 3-5 次同类任务，30天后自动归档        │
├─────────────────────────────────────────┤
│ L2: 长期准则 (AGENTS.md 精简版)          │
│ 只存"原则"，不存"细节"                   │
├─────────────────────────────────────────┤
│ L3: 完整归档 (memory/lessons/archive/)   │
│ 完整复盘记录，按需查询，不常驻内存         │
└─────────────────────────────────────────┘
```

#### 存储规则

| 层级 | 内容 | 生命周期 | 触发条件 |
|------|------|---------|---------|
| L0 | 当前任务调整 | 单次会话 | 每次任务 |
| L1 | 详细经验记录 | 30天 | 价值评分 ≥ 50 |
| L2 | 抽象原则 | 永久 | 价值评分 ≥ 70 |
| L3 | 完整复盘 | 永久 | 需要时查询 |

#### AGENTS.md 存储格式（只存原则）

```markdown
## DCE 经验原则 (Auto-generated)

### 配置修改类
- 必须检查: 备份、影响范围、回滚方案
- 常见遗漏: 依赖服务、环境变量

### 文案创作类
- 必须先确认: 目标受众、平台特性、风格偏好
- 常见偏差: 过于正式/随意、忽略平台算法
```

**不存的内容**:
- ❌ "上次用户选了方案B因为..."
- ❌ "3月5日的任务中..."
- ❌ 具体对话记录

### 2.3 学习层 (Layer 3) - Learning Loop

#### 两种复盘机制

**类型一：即时复盘（讨论中）**

- **触发**: 方案大幅调整，引入新信息
- **目的**: 学习用户习惯，下次减少讨论
- **存储**: L0 → L1
- **升级条件**: 同类任务出现 3+ 次

**类型二：执行后复盘（效果验证）**

- **触发**: 执行效果与预期不符
- **目的**: 深度反思，找出缺失信息
- **存储**: L1 → L2/L3
- **升级条件**: 价值评分 ≥ 70

#### 复盘模板

```markdown
## DCE 复盘: [任务类型]

### 原始方案
- 方案 A: ...
- 方案 B: ...

### 最终选择
指挥官选择了: 方案 X

### 执行偏差
- 预期: ...
- 实际: ...

### 经验教训
1. D 阶段遗漏了 [某维度]
2. C 阶段的确认可以更 [某方式]
3. 下次同类任务应该 [某优化]

### 价值评分
- 复用频率: X/5
- 失败避免: X/5
- 效率提升: X/5
- 认知成本: X/5
- 总分: XX (加权)

### 存储决策
- [ ] 升级至 AGENTS.md (≥70分)
- [ ] 保留在 L1 (50-69分)
- [ ] 归档至 L3 (<50分)
```

## 三、主动信息挖掘机制

### 3.1 触发条件

当用户回应包含以下模式时，进入"信息挖掘模式":

| 模式 | 含义 | Agent 行动 |
|------|------|-----------|
| "我觉得..." | 新观点 | 深挖背后原因 |
| "但是..." | 有顾虑 | 澄清顾虑点 |
| "能不能..." | 新需求 | 扩展需求范围 |
| "如果..." | 假设场景 | 分析边界条件 |
| 方案大幅调整 | 信息不足 | 系统性追问 |

### 3.2 信息挖掘框架 (5W2H + 极限)

| 维度 | 问题 | 目的 |
|------|------|------|
| **Why** | 为什么要做这个？核心目标？ | 对齐优先级 |
| **What** | 具体要做什么？范围边界？ | 明确交付物 |
| **Who** | 受众是谁？决策者是谁？ | 定位风格/权限 |
| **When** | 截止时间？关键节点？ | 评估可行性 |
| **Where** | 平台/场景/环境？ | 确定约束条件 |
| **How** | 期望的方式？有参考案例？ | 对齐执行方式 |
| **How much** | 预算/资源/量级？ | 评估方案可行性 |
| **极限场景** | 最坏情况？异常处理？ | 确保鲁棒性 |

### 3.3 渐进明细流程

```
用户提出任务
    ↓
Agent 初步分析 → 给出 3 个方案
    ↓
用户回应
    ├── 直接选择 → 进入 C 阶段
    ├── 简单调整 → 优化方案 → 进入 C 阶段
    └── 引入新信息 → 触发【信息挖掘模式】
              ↓
        Agent 分析新信息
              ↓
        追问相关问题（5W2H + 极限）
              ↓
        补充信息足够？
              ├── 否 → 继续追问（最多 3 轮）
              └── 是 → 重新生成方案 → 进入 C 阶段
```

### 3.4 防 Token 爆炸机制

| 机制 | 规则 | 目的 |
|------|------|------|
| 轮次上限 | 最多 3 轮追问 | 防止无限讨论 |
| 问题上限 | 每轮最多 3 个问题 | 控制单次成本 |
| 预算提醒 | 达到 80% Token 预算时提醒 | 让用户知情 |
| 快速通道 | 用户说"简单处理"跳过挖掘 | 紧急情况 |
| 分阶段建议 | 信息严重不足时建议拆分任务 | 降低复杂度 |

## 四、效果与成本评估

### 4.1 核心指标

| 指标 | 定义 | 目标 |
|------|------|------|
| 命中率 | 首次方案被接受的比例 | > 60% |
| 完善度 | 1 - (遗漏问题数 / 总问题数) | > 80% |
| 返工率 | 执行后需要重新调整的比例 | < 20% |
| 用户满意度 | 主观评分 1-5 | > 4.0 |

### 4.2 Token 成本预估

| 阶段 | 传统方式 | DCE 2.0 | 变化 |
|------|---------|---------|------|
| 初始方案 | 1 轮 | 1 轮 | 相同 |
| 讨论调整 | 1-3 轮（被动） | 2-4 轮（主动） | +20-50% |
| 执行后修复 | 经常有 | 大幅减少 | -50-80% |
| **总成本趋势** | 100% | 短期 120% → 长期 60% | 先增后减 |

### 4.3 v1.0 简化评估

只记录基础数据：
```markdown
## DCE 执行记录
- 任务类型: X
- 讨论轮次: Y
- Token 消耗: Z
- 是否返工: [是/否]
```

复杂分析（对比实验、满意度评分）延后至 v2.0。

## 五、触发条件清单

### 5.1 必须使用 DCE 的场景

| 场景 | 例子 | 风险等级 |
|------|------|---------|
| 配置修改 | 改 openclaw.json、Gateway 配置 | 🔴 高 |
| 外部通信 | 发邮件、发微博、发消息 | 🔴 高 |
| 数据删除 | 删文件、清记录、移除权限 | 🔴 高 |
| 安全相关 | 加用户、改权限、装插件 | 🔴 高 |
| 不可逆操作 | 发布、部署、转账 | 🔴 高 |
| 资源消耗 | 大量 Token、长时间任务 | 🟡 中 |
| 首次任务 | 新类型、新领域 | 🟡 中 |

### 5.2 快速通道（跳过 DCE）

- 用户明确说"简单处理"
- 用户说"按常规/上次方式"
- 重复任务且历史记录良好
- 紧急任务（事后补复盘）

## 七、开源战略路线图

### 核心原则

> **"卖方法比卖工具更有价值，但工具是方法的最佳广告。"**

### 三阶段路线

| 阶段 | 内容 | 时间 |
|------|------|------|
| 一 | `organization-mastery` 理论库 | 现在 |
| 二 | `agent-skills` 实践库 | 验证后 |
| 三 | 生态建设，双向引用 | 长期 |

### 先做理论后做工具的原因

1. **理论是根，工具是叶**
2. **理论护城河更深**（难以复制）
3. **受众更有价值**（团队负责人 > 个人开发者）
4. **工具是理论的广告**

### 10.2 快速参考卡

```
DCE 三步走：
1. Discuss: 给方案 → 挖信息 → 再优化
2. Confirm: 等"执行"，不猜不蒙
3. Execute: 先确认 → 再行动 → 后复盘

信息挖掘 5W2H：
Why What Who When Where How How-much

经验存储：
L0(临时) → L1(短期) → L2(原则) → L3(归档)
```
